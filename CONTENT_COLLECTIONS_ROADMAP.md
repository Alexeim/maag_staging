# Дорожная карта по Content Collections

## Зачем нужен этот документ

Фича `contentCollections` сама по себе правильная, но она одновременно затрагивает три чувствительных слоя:

- редакторы и криэйторы в дашборде
- загрузку и сохранение на фронтенде
- правила синхронизации в Firestore

В проекте уже есть утверждённый паттерн ленивой загрузки Alpine через `src/lib/alpine/plugins/lazyLoadPlugin.ts` и `src/alpine-entrypoint.ts`. Если внедрять `contentCollections` в обход этого паттерна, мы получим дублирующиеся пути инициализации, более сложную отладку и хрупкое поведение редакторов.

Этот документ нужен, чтобы зафиксировать контракт до следующих изменений в коде.

## Цель

Добавить `contentCollections` как переиспользуемый редакторский слой связки контента, который будет работать во всех криэйторах и редакторах и поддерживать:

- выбор существующей коллекции
- создание новой коллекции прямо из редактора
- двустороннее сохранение принадлежности к коллекции
- дальнейшее чтение связанного контента на публичной странице материала

`relatedContent` остаётся как ручной однонаправленный способ связки.
`contentCollections` становится структурированным двусторонним и групповым слоем связей.

## Подтверждённые продуктовые правила

### Документ материала

Каждый контент-документ должен хранить:

```ts
contentCollectionId: string | null
```

Это относится к:

- article
- event
- flipper
- guide
- interview
- news
- visualStory

### Документ коллекции

Коллекция Firestore: `contentCollections`

Структура документа:

```ts
{
  id: string;
  title: string;
  content: {
    article: string[];
    event: string[];
    flipper: string[];
    guide: string[];
    interview: string[];
    news: string[];
    visualStory: string[];
  };
  createdAt: Date;
  updatedAt: Date;
}
```

### Правила синхронизации

Когда материал добавляется в коллекцию:

- записать `collectionId` в `material.contentCollectionId`
- добавить `materialId` в `contentCollections/{collectionId}.content[type]`

Когда материал удаляется из коллекции:

- сбросить `material.contentCollectionId` в `null`
- удалить `materialId` из `contentCollections/{collectionId}.content[type]`

Когда коллекция создаётся прямо из редактора:

- создать `contentCollections/{id}`
- сохранить `title`
- инициализировать все массивы `content.*`
- сразу вставить текущий материал в массив нужного типа
- сразу записать новый `id` в `material.contentCollectionId`

## Текущее состояние кодовой базы

### Существующие фронтенд-паттерны

- Редакторы уже используют ленивую загрузку Alpine через `$lazy(...)`.
- Логика криэйторов разделена по нескольким модулям:
  - `src/components/article/creatorLogic.ts`
  - `src/components/article/eventCreatorLogic.ts`
  - `src/components/article/guideCreatorLogic.ts`
  - `src/components/article/visualStoryCreatorLogic.ts`
  - `src/components/dashboard/flipperCreatorLogic.ts`
  - `src/components/dashboard/interviewCreatorLogic.ts`
  - `src/components/dashboard/newsCreatorLogic.ts`
  - `src/components/dashboard/tipsArticleCreatorLogic.ts`
- Для `relatedContent` уже есть переиспользуемый UI-блок:
  - `src/components/dashboard/RelatedContentEditor.astro`
- `authors` уже дают наиболее близкий UX-референс:
  - список существующих сущностей
  - создание новой прямо в редакторе, если нужной нет

### Архитектурное ограничение

Нельзя добавлять второй путь загрузки Alpine.

Это значит:

- никакого ручного `Alpine.start()`
- никакой регистрации компонентов через случайные скрипты вне текущего lazy-loading flow
- никакой инициализации фичи в обход `$lazy`

`contentCollections` должны встраиваться в существующие модули логики криэйторов и в переиспользуемые Astro-блоки.

## Рекомендуемая стратегия реализации

## Фаза 1: Зафиксировать общий контракт коллекций

Нужно создать один общий utility-модуль для типов коллекций и нормализации данных.

Предлагаемый новый utility на фронтенде:

```ts
src/lib/utils/contentCollections.ts
```

Ответственность:

- определить поддерживаемые типы контента
- возвращать пустую структуру `content` для коллекции
- нормализовывать `contentCollectionId`
- нормализовывать документы коллекций, приходящие из API

Предлагаемый общий utility на сервере:

```ts
server/src/utils/contentCollections.ts
```

Ответственность:

- нормализовывать название коллекции
- собирать пустую типизированную карту `content`
- валидировать входящий payload коллекции
- удалять дубликаты id

Причина:
Без общей нормализации каждый криэйтор начнёт жить своей версией правил.

## Фаза 2: Добавить отдельный API для коллекций

Нужно идти по тому же паттерну, что и `authors`.

Предлагаемые endpoints:

- `GET /api/content-collections`
- `POST /api/content-collections`
- `GET /api/content-collections/:id`
- `PUT /api/content-collections/:id`

Опционально позже:

- `DELETE /api/content-collections/:id`

Минимальная форма payload для API:

```ts
type ContentCollectionPayload = {
  title: string;
};
```

Сервер должен взять на себя:

- генерацию id документа, когда это нужно
- инициализацию пустых типизированных массивов
- возврат нормализованного ответа

Важно:
Нельзя позволять фронтенду напрямую писать membership в Firestore.
Логику синхронизации нужно держать централизованно на сервере.

## Фаза 3: Добавить синхронизацию коллекций в потоки сохранения контента

Это самое важное серверное решение во всей фиче.

Для каждого контроллера создания и обновления:

- article
- event
- flipper
- guide
- interview
- news
- visual story

поток сохранения должен сравнивать:

- предыдущее `contentCollectionId`
- новое `contentCollectionId`

Дальше он должен:

- удалить id материала из старой коллекции, если связь изменилась
- добавить id материала в новую коллекцию, если она выбрана
- синхронизировать поле в самом документе материала

Причина:
Только сервер может надёжно обеспечивать двустороннюю консистентность.

## Фаза 4: Добавить переиспользуемый UI-блок в дашборде

Нужно создать отдельный UI-блок, не смешивая его с `RelatedContentEditor.astro`.

Предлагаемый новый компонент:

```ts
src/components/dashboard/ContentCollectionsEditor.astro
```

Почему отдельно:

- `relatedContent` типизируется по виду привязанного контента
- `contentCollections` описывает принадлежность к коллекции
- по UX это ближе к `authors`, чем к `relatedContent`

Предлагаемое поведение UI:

- показывать текущую выбранную коллекцию
- давать выбрать одну существующую коллекцию из dropdown
- уметь переключаться в режим создания новой коллекции по title
- назначать выбранную коллекцию
- создавать новую коллекцию и сразу назначать её
- снимать текущую коллекцию

Предлагаемое Alpine-state в криэйторах:

```ts
availableContentCollections: Array<{ id: string; title: string }>;
contentCollectionsLoading: boolean;
contentCollection: { id: string; title: string } | null;
selectedContentCollectionId: string;
useNewContentCollection: boolean;
newContentCollectionTitle: string;
```

В сам payload материала должно попадать только:

```ts
contentCollectionId: string | null
```

## Фаза 5: Переиспользовать общий паттерн интеграции криэйторов

Не нужно строить collection-логику отдельно внутри каждого Astro-файла.

Вместо этого нужно:

- добавить общие helper-методы для загрузки, создания, выбора, замены и снятия коллекции
- подключить их в каждый модуль логики криэйтора
- монтировать один и тот же `ContentCollectionsEditor.astro` во всех composer-компонентах

Лучшее направление:

- вынести общие helper-методы редактора коллекций в переиспользуемый модуль
- импортировать эти helper-методы в разные creator-logic файлы

Причина:
В кодовой базе уже много дублирования между криэйторами.
Если просто копировать collection-логику семь раз, стоимость поддержки вырастет сразу.

## Фаза 6: Использование на публичных страницах

Публичные страницы должны читать `contentCollectionId` из текущего контент-документа и использовать этот id для загрузки нужной коллекции.

Дальше нужно:

- собрать id из соответствующих `content[type]` массивов
- убрать id текущего материала
- разрешить эти id в реальные сущности контента

Эту часть можно делать после того, как редакторский поток станет стабильным.
Не нужно блокировать редакторскую реализацию из-за публичного рендера.

## Что, скорее всего, пошло не так в прошлой реализации

Судя по текущей структуре проекта, самые рискованные зоны были такими:

### 1. Обход существующего контракта Alpine lazy-loader

Если логика цеплялась через кастомные скрипты или альтернативные пути регистрации, фича начинала вести себя случайно и хрупко при переходах между страницами.

### 2. Смешивание трёх задач в один шаг

Во фиче есть три отдельные задачи:

- CRUD для коллекций
- синхронизация membership
- UX редактора

Если делать всё это сразу без заранее зафиксированного контракта, результат становится трудно понимать и ещё труднее поддерживать.

### 3. Попытка обращаться с коллекциями как с `relatedContent`

Связь есть, но это не одно и то же:

- `relatedContent` хранит прямые типизированные id внутри материала
- `contentCollections` хранит групповую принадлежность и требует двусторонней синхронизации

UI местами может быть похож, но логика сохранения у этих вещей принципиально разная.

## Что не входит в первый проход

Пока не нужно трогать это, пока базовый flow не станет стабильным:

- удаление коллекции через UI
- переименование коллекции через UI
- порядок или ранжирование внутри коллекции
- концепцию primary collection
- автоматическую миграцию из `relatedContent`
- эвристику рекомендаций на публичной странице

## Рекомендуемый порядок доставки

1. Добавить общие schema utilities для коллекций на фронтенде и сервере.
2. Добавить endpoints `content-collections` и response types.
3. Добавить `contentCollectionId` во фронтендовые payload types.
4. Добавить серверную sync-логику в create/update flows.
5. Добавить переиспользуемый `ContentCollectionsEditor.astro`.
6. Сначала встроить блок только в один криэйтор, лучше всего в `article`.
7. Проверить сценарии create, edit, remove и inline-create.
8. После этого раскатать тот же паттерн на остальные криэйторы.
9. И только потом подключать использование коллекций на публичных страницах.

## Критерии готовности редакторского потока

Первый проход можно считать завершённым, когда:

- редактор умеет выбрать существующую коллекцию
- редактор умеет создать новую коллекцию прямо внутри формы
- сохранение пишет `contentCollectionId` в документ материала
- сохранение пишет id материала в `contentCollections.content[type]`
- удаление коллекции обновляет обе стороны
- edit mode заранее подгружает уже связанную коллекцию
- решение работает через существующий Alpine `$lazy` flow
- в проект не добавлен ручной feature-specific bootstrapping Alpine

## Практическая рекомендация перед следующим кодом

Не нужно сразу снова бросаться в реализацию.

Следующий шаг должен быть таким:

1. Просмотреть этот roadmap.
2. Подтвердить API-контракт и первый тип контента для внедрения.
3. Реализовать фичу вертикально только для одного типа контента.
4. И только потом масштабировать её на остальные криэйторы.

Так мы не повторим тот же интеграционный бардак ещё раз.
